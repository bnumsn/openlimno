+## Three residuals shipped as known limitations
+
+These are real concerns, not P0/P1-tagged by any reviewer, documented
+here for v0.1.0 final:
+
+1. **Missing-backend retry futility** (Gemini #2 + Claude #1, consensus)
+   `_read_wua_parquet` raises `RuntimeError` when neither pyarrow nor
+   GDAL is importable. The controller's retry loop catches `RuntimeError`
+   as transient and burns 3 attempts × 50 ms backoff before giving up.
+   Cosmetic latency on a guaranteed-failure path; no data corruption.
+   **Filed as issue:** consider a `MissingBackendError` subclass that the
+   controller short-circuits on.
+
+2. **GUI direct-call paths bypass the cache wrapper** (Codex P2)
+   `Controller.open_wua_q` and `plot_cross_section` call
+   `_read_wua_parquet` directly rather than `_read_xs_rows_cached`.
+   Round-7 changed the helper to propagate read failures, so a torn
+   parquet now raises through to the GUI handler. Acceptable for an
+   alpha — Qt's default exception handler shows a dialog rather than
+   crashing — but the cache wrapper should be the single entry point.
+
+3. **OGR fallback resource hygiene** (Claude #3)
+   The GDAL/OGR branch never explicitly closes the `Dataset`, and
+   `GetLayer(0)` returning `None` on malformed input crashes with
+   `AttributeError` (not retried under round-7's narrowed exception
+   list). Long-running GUI sessions could leak file handles. Mitigation:
+   pyarrow is the bundled-default path; OGR is the fallback that few
+   users will hit in practice.
+

+
+Concerns, in priority order:
+
+**1. `RuntimeError` retried for permanent config failure (controller.py:881)**
+The refactored `_read_wua_parquet` raises `RuntimeError("neither pyarrow nor GDAL available...")` when both backends are missing. The retry loop now catches `RuntimeError` and burns the full retry budget (with `time.sleep` backoff) on what is a permanent install problem. Should special-case missing-backend (or use a distinct exception type) and propagate immediately.
+
+**2. The `ArrowInvalid` claim looks wrong (controller.py:880-887 comment)**
+The comment asserts "ArrowInvalid is a subclass of OSError in newer pyarrow, ValueError in older." In reality `pyarrow.lib.ArrowInvalid` inherits from `pyarrow.lib.ArrowException` → `Exception` in essentially every released pyarrow. Neither `OSError` nor `ValueError` catches it. So the very corruption case the change exists to handle (torn parquet footer) likely escapes the new `except` list and surfaces as an uncaught crash. Verify against the bundled pyarrow version and add `pyarrow.lib.ArrowException` (or import-guarded equivalent) to the `except` tuple. Without this fix the cache-bypass-on-torn-read goal is not met.
+
+**3. OGR fallback resource leaks and None-layer crash (controller.py:_read_wua_parquet GDAL branch)**
+`ds = ogr.Open(path)` is never closed; in a long-running PyQt GUI session this leaks file handles. `ds.GetLayer(0)` may return `None` for malformed datasets — `defn = layer.GetLayerDefn()` then raises `AttributeError`, which the new exception-narrowing classifies as a "programmer bug" and refuses to retry, even though it is actually corrupt-input. Wrap in `try/finally` and explicit None-check that raises `OSError`.
+
+**4. Test coverage is thin (tests/unit/test_xs_rows_cache.py)**
+The added `test_typeerror_propagates_does_not_retry` only patches `_read_wua_parquet` to raise `TypeError`. The change comment names `TypeError, AttributeError, KeyError` — only one is exercised. Also missing:
+- A test that a real torn parquet file (bad footer / truncated bytes) actually hits the retry path through the *unmocked* `_read_wua_parquet`. Mocking sidesteps issue #2 entirely; a regression where ArrowException leaks past the `except` would still pass these tests.
+- A test that the missing-backend `RuntimeError` does NOT retry (it currently does — issue #1).
+- A test that `ValueError` / `OSError` / `EOFError` each trigger a retry (positive case).
+A regression in the new exception list would not be caught.
+
+**5. Shell `echo` portability (scripts/triple_review.sh:46)**
+`base_lower="$(echo "$BASE" | tr 'A-F' 'a-f')"`: if `BASE` happens to be `-n` or contains backslashes, `echo` behavior is shell-dependent (bash builtin vs `/bin/echo` vs dash). Use `printf '%s' "$BASE" | tr ...`. Minor on GitHub Actions today but not portable.
+
+**6. Hex-named branches still bypass `check-ref-format`**
+`^[0-9a-f]{8,40}$` accepts e.g. `deadbeef`, `feedface`, `cafebabe1` as "SHA" without ever reaching the ref-format fallback. If such branches exist locally, git resolution rules apply silently. +
+**7. Architecture nit — abstraction churn**
+Five rounds of review-driven exception-list re-narrowing on the same 6-line `except` block is itself a smell. +

+# Gemini review — `v0.1.0-alpha.9..86ff7a7`
+
+_Generated by `gemini -p` on 2026-05-09T13:28:40+08:00_
+
+### 1. Architecture and Design
+The narrowing of exception handling in `src/openlimno/gui_core/controller.py` aligns with robust systems programming by distinguishing between transient I/O failures and programmer errors. However, the implementation of the fallback logic in `_read_wua_parquet` (lines 58–61) introduces a minor architectural flaw: it raises a `RuntimeError` when dependencies are missing, which the caller (`Controller._read_xs_rows_cached`, line 881) then catches and retries. Retrying a missing library installation is futile and adds unnecessary latency before the eventual failure.
+
+### 2. Bug Risk
+*   **Ref Validation (`scripts/triple_review.sh:44`):** The logic explicitly rejects any 7-character hex string as a "collision-prone SHA." While this protects against Git's default short-SHA length, it creates a false positive for legitimate branch or tag names that happen to be 7-character hex strings (e.g., `beef123`, `cafebad`). If a user has a naming convention involving hex prefixes, this script will block their refs.
+*   **Parquet Reader Fallback (`src/openlimno/gui_core/controller.py:53`):** The OGR backend check uses `if ds is None` but then proceeds to call `ds.GetLayer(0)`. While the code now raises `OSError` if `ds` is `None` (line 72), the error message "GDAL/OGR failed to open..." is generic. It could be a missing file, a permission issue, or an unsupported parquet flavor (e.g., encrypted), yet it is treated as a transient retryable error in the controller.
+*   **Exception Narrowing (`src/openlimno/gui_core/controller.py:881`):** You catch `ValueError` to handle `ArrowInvalid` on older pyarrow versions. However, `ValueError` is a very broad built-in exception. If `_read_wua_parquet` or any logic inside the `try` block (like `zip(..., strict=False)` if it were to fail) raises a `ValueError` due to a logic bug, the controller will incorrectly treat it as a transient I/O issue and burn the retry budget.
+
+### 3. Performance
+*   **Row Conversion (`src/openlimno/gui_core/controller.py:46`):** The pyarrow path converts every column to a Python list (`c.to_pylist()`) and then zips them. For large hydraulic datasets, this is highly memory-inefficient as it duplicates the entire table in memory before creating the list of dicts. `table.to_pydict()` followed by a transformation or using `to_batches()` would be significantly more performant.
+*   **GUI Blocking:** This code resides in `gui_core`. If `_read_xs_rows_cached` is called on the main thread, the combination of multiple I/O retries and `time.sleep` (implied by `mock_sleep` in tests) will hang the UI.
+
+### 4. Test Adequacy
+*   **Missing Dependency Path:** There is no test verifying the behavior when *neither* `pyarrow` nor `osgeo` is installed. As noted in section 1, this currently triggers an unnecessary retry loop.
+*   **Torn Read Recovery:** The tests mock the exception but don't verify that the *second* attempt (after a simulated transient failure) actually succeeds and returns correct data.
+*   **GDAL Path:** The new test only mocks the pyarrow entry point. There is no coverage for the OGR fallback path or its specific failure modes (e.g., `ds is None`).
+
+### 5. Packaging and Portability
+*   **Dependency Assumptions:** The `pyproject.toml` bump is standard, but the code assumes `osgeo` (GDAL) might be available as a fallback. GDAL is notoriously difficult to install via `pip` without external C libraries. If the AppImage/distribution strategy (implied by `packaging/` directory) relies on this fallback, it needs rigorous validation on the target runners.
+
+### 6. Recommendations
+1.  **Refine Retry Logic:** In `_read_wua_parquet`, raise a specific `DependencyError` (subclass of `Exception`, not `RuntimeError`) when libraries are missing, and ensure `Controller` does **not** catch/retry that specific type.
+2.  **Optimize Parquet Read:** Use `t.to_batches()` to iterate rows if memory pressure is a concern.
+3.  **Ref Validation:** Instead of a hard block on 7-char hex, consider checking if the ref exists via `git rev-parse --verify "$BASE^{commit}"` to disambiguate.

## How to read the output

Open all three review files **before triaging anything**. The pattern that
matters is the *intersection* and *symmetric difference* of findings:

- A concern raised by **all three** → almost certainly real, fix it.
- A concern raised by **two** → likely real, check it carefully.
- A concern raised by **only one** → could be the load-bearing find of the
  round, OR could be a false positive. Read closely. Don't dismiss.
- **No concerns from any** → suspicious. Either the diff is genuinely
  trivial, or the prompt scope was too narrow.

The summary file (`reviews/<sha>.summary.md`) lists which files each
reviewer touched, but does NOT consolidate findings. That's deliberate:
a meta-AI summary would re-introduce the single-reviewer bias the whole
process is designed to avoid.

2. **Defer with rationale** — file an issue and link it from the next
   release notes. Don't silently ignore.
3. **Reject with rationale** — same: write the rationale somewhere durable
   so it doesn't re-surface.
